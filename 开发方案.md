# Windows 本地游戏管理平台技术方案

# 1. 项目目标

开发一款运行在 Windows 上的本地游戏管理平台，带 UI 界面，功能类似 Steam 的本地游戏库，但主要面向无需联网的本地游戏。

核心功能包括：

1. 展示本地游戏列表  
   - 每个游戏选项由游戏封面图片和游戏名称组成。
   - 游戏名称显示在封面图片下方。
   - 展示形式类似购物平台商品列表或 Steam 游戏库。

2. 查看游戏详情  
   - 点击游戏选项后进入游戏详情界面。
   - 显示游戏封面、游戏名称、总游玩时长、上次启动日期等信息。

3. 启动本地游戏  
   - 在游戏详情页点击“开始游戏”按钮。
   - 启动本地电脑中的游戏 EXE 文件。
   - 自动统计本次游玩时长。
   - 游戏关闭后更新总游玩时长和上次启动时间。

4. 添加本地游戏  
   - 用户可以手动添加本地游戏。
   - 添加时需要选择：
     - 游戏启动文件
     - 游戏存放目录
     - 游戏存档目录
     - 游戏封面图片
     - 游戏名称

5. 使用 WebDAV 同步用户数据  
   - 参考 Kazumi 项目使用 WebDAV 同步用户个人数据的思路。
   - 通过 WebDAV 保存和同步：
     - 游戏封面
     - 游戏名称
     - 游玩时长
     - 上次启动时间
     - 游戏存档路径信息
     - 游戏存档压缩包
   - 首期可以优先支持坚果云 WebDAV。
   - 后续可以扩展支持 Nextcloud、群晖、Alist 等其他 WebDAV 服务。

# 2. 推荐技术选型

## 2.1 首选方案

如果项目只面向 Windows 平台，推荐使用：

```text
C# / .NET 8 + WPF 或 WinUI 3
```

这是最适合 Windows 本地游戏管理器的技术组合。

原因是本项目有大量 Windows 桌面端能力需求，例如：

- 选择本地 EXE 文件
- 启动本地游戏进程
- 监听游戏进程退出
- 访问游戏目录
- 访问存档目录
- 监控文件变化
- 压缩和解压存档
- 保存本地配置
- 使用系统文件选择器
- 使用 Windows 凭据管理器保存 WebDAV 密码
- 打包为 Windows 安装程序

这些能力使用 C#/.NET 会比 Flutter、Electron 更直接、更稳定。

------

## 2.2 具体技术栈

| 模块         | 推荐技术                                        |
| ------------ | ----------------------------------------------- |
| 编程语言     | C#                                              |

| UI 框架      | WPF 或 WinUI 3                                  |

| 打包发布     | MSIX / Inno Setup / Squirrel.Windows            |

------

## 2.3 WPF 和 WinUI 3 的选择

### WPF

优点：

- 成熟稳定
- 资料多
- 第三方控件多
- 适合传统 Windows 桌面软件
- 对 MVVM 支持非常成熟
- 开发效率高
- .NET 8 仍然支持良好

缺点：

- UI 风格偏传统
- 默认控件视觉效果不如 WinUI 3 现代
- 如果想做非常现代的 UI，需要额外美化

推荐场景：

```text
如果你是第一次开发 Windows 桌面应用，或者希望快速做出稳定版本，推荐 WPF。
```

------

### WinUI 3

优点：

- UI 更现代
- 更接近 Windows 11 风格
- 原生支持现代控件
- 适合做漂亮的桌面应用

缺点：

- 生态和资料不如 WPF 成熟
- 部分功能开发复杂度更高
- 打包和部署比 WPF 稍麻烦

推荐场景：

```text
如果你非常重视界面现代感，并且愿意接受更高的学习成本，可以选择 WinUI 3。
```

------

## 2.4 最终建议

第一版推荐：

```text
C# + .NET 8 + WPF + SQLite + WebDAV
```

原因：

- 开发速度快
- 稳定性高
- 文档和资料多
- 适合本地文件管理和进程管理
- 更适合先做 MVP

等功能稳定后，如果想提升 UI 质感，可以：

- 使用 WPF UI
- 使用 MaterialDesignInXaml
- 使用 HandyControl
- 或者后期迁移到 WinUI 3

# 3. 总体架构设计

项目整体采用“本地优先 + 云同步”的设计。

也就是说：

```text
所有功能优先使用本地数据库和本地文件缓存。
WebDAV 只负责同步和备份，不作为实时数据库使用。
```

这样做的好处：

- 软件离线也能正常使用
- 游戏启动不依赖网络
- UI 加载速度快
- WebDAV 失败不会影响基本功能
- 数据更安全
- 后续可以支持更多云服务

# 4. 项目分层设计

## 4.1 UI 层

负责显示界面，不直接操作数据库、不直接操作 WebDAV、不直接启动游戏。

主要页面：

```text
GameLibraryView      游戏库首页
GameDetailView       游戏详情页
AddGameView          添加游戏页
EditGameView         编辑游戏页
SettingsView         设置页
SyncLogView          同步日志页
```

UI 层只和 ViewModel 通信。

------

## 4.2 ViewModel 层

负责 UI 状态管理和命令绑定。

例如：

```text
GameLibraryViewModel
- Games
- SearchText
- LoadGamesCommand
- OpenGameDetailCommand
- AddGameCommand

GameDetailViewModel
- CurrentGame
- TotalPlayTimeText
- LastLaunchTimeText
- StartGameCommand
- SyncSaveCommand
- EditGameCommand
```

ViewModel 不直接访问 SQLite 和 WebDAV，而是调用 Service。

------

## 4.3 应用服务层

这是项目的核心业务逻辑层。

主要服务：

```text
GameService

SaveSyncService
WebDavSyncService
CoverService

```

------

### GameService

负责游戏信息管理。

功能：

- 添加游戏
- 删除游戏
- 编辑游戏
- 获取游戏列表
- 获取游戏详情
- 更新游戏封面
- 更新游戏路径
- 更新游戏存档目录

------

### LaunchService

负责启动游戏。

功能：

- 根据游戏配置启动 EXE
- 设置启动参数
- 设置工作目录
- 监听进程退出
- 处理启动失败
- 处理启动器和实际游戏进程分离的问题

------

### PlayTimeService

负责游玩时长统计。

功能：

- 记录游戏启动时间
- 记录游戏退出时间
- 计算本次游玩时长
- 更新总游玩时长
- 生成游玩记录
- 合并多设备游玩记录

------

### SaveSyncService

负责存档同步。

功能：

- 扫描存档目录
- 计算文件哈希
- 判断存档是否发生变化
- 打包存档目录
- 解压云端存档
- 上传存档压缩包
- 下载存档压缩包
- 恢复存档前自动备份
- 处理本地和云端冲突

------

### WebDavSyncService

负责 WebDAV 云端同步。

功能：

- 测试 WebDAV 连接
- 创建远程目录
- 上传 metadata.json
- 下载 metadata.json
- 上传 cover.jpg
- 下载 cover.jpg
- 上传 latest.zip
- 下载 latest.zip
- 获取远程文件信息
- 判断远程文件是否存在

------

### CoverService

负责游戏封面处理。

功能：

- 复制用户选择的封面图片
- 生成本地封面缓存
- 上传封面到 WebDAV
- 从 WebDAV 下载封面
- 生成缩略图
- 处理封面缺失情况

------

### SettingsService

负责应用设置。

功能：

- 保存 WebDAV 地址
- 保存 WebDAV 用户名
- 保存 WebDAV 密码或应用密码
- 保存同步根目录
- 保存自动同步开关
- 保存 UI 设置
- 保存本机 ID

------

### BackupService

负责本地备份。

功能：

- 覆盖存档前备份
- 上传云端前备份
- 保存历史存档
- 清理过旧备份

------

### ConflictService

负责同步冲突判断。

功能：

- 判断本地和云端哪个更新
- 判断是否有双端修改
- 生成冲突提示
- 执行用户选择的冲突解决方案

# 7. SQLite 数据库设计

## 7.1 games 表

用于保存游戏基本信息。

```sql
CREATE TABLE games (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  executable_path TEXT NOT NULL,
  game_root_path TEXT NOT NULL,
  save_path TEXT NOT NULL,
  cover_image_path TEXT,
  launch_args TEXT,
  working_directory TEXT,
  monitor_process_name TEXT,
  total_play_seconds INTEGER DEFAULT 0,
  last_launch_time TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  cloud_revision TEXT,
  sync_enabled INTEGER DEFAULT 1
);
```

字段说明：

| 字段                 | 说明                   |
| -------------------- | ---------------------- |
| id                   | 游戏唯一 ID，使用 UUID |
| name                 | 游戏名称               |
| executable_path      | 游戏启动 EXE 路径      |
| game_root_path       | 游戏安装目录           |
| save_path            | 游戏存档目录           |
| cover_image_path     | 本地封面图片路径       |
| launch_args          | 启动参数               |
| working_directory    | 工作目录               |
| monitor_process_name | 实际监控的游戏进程名   |
| total_play_seconds   | 总游玩时长，单位为秒   |
| last_launch_time     | 上次启动时间           |
| created_at           | 创建时间               |
| updated_at           | 更新时间               |
| cloud_revision       | 云端版本标识           |
| sync_enabled         | 是否开启同步           |

------

## 7.2 play_sessions 表

用于保存每一次游玩记录。

```sql
CREATE TABLE play_sessions (
  id TEXT PRIMARY KEY,
  game_id TEXT NOT NULL,
  machine_id TEXT NOT NULL,
  start_time TEXT NOT NULL,
  end_time TEXT,
  duration_seconds INTEGER,
  exit_code INTEGER,
  synced INTEGER DEFAULT 0,
  created_at TEXT NOT NULL,
  FOREIGN KEY(game_id) REFERENCES games(id)
);
```

字段说明：

| 字段             | 说明             |
| ---------------- | ---------------- |
| id               | 本次游玩记录 ID  |
| game_id          | 对应游戏 ID      |
| machine_id       | 本机设备 ID      |
| start_time       | 启动时间         |
| end_time         | 退出时间         |
| duration_seconds | 本次游玩时长     |
| exit_code        | 游戏进程退出码   |
| synced           | 是否已同步到云端 |
| created_at       | 创建时间         |

使用 `play_sessions` 的好处是：

```text
不要只同步 total_play_seconds。
应该同步每一次游玩记录，然后通过所有记录求和得到总游玩时长。
```

这样多台电脑之间同步时，不容易互相覆盖游玩时长。

------

## 7.3 sync_records 表

用于记录同步日志。

```sql
CREATE TABLE sync_records (
  id TEXT PRIMARY KEY,
  game_id TEXT,
  sync_type TEXT NOT NULL,
  direction TEXT NOT NULL,
  status TEXT NOT NULL,
  started_at TEXT NOT NULL,
  finished_at TEXT,
  message TEXT,
  FOREIGN KEY(game_id) REFERENCES games(id)
);
```

字段说明：

| 字段        | 说明                               |
| ----------- | ---------------------------------- |
| id          | 同步记录 ID                        |
| game_id     | 对应游戏 ID，可以为空              |
| sync_type   | 同步类型，如 metadata、cover、save |
| direction   | 同步方向，如 upload、download      |
| status      | 状态，如 success、failed、conflict |
| started_at  | 开始时间                           |
| finished_at | 完成时间                           |
| message     | 错误信息或备注                     |

## 9.5 本机路径文件


# 10. 游戏启动和游玩时长统计设计

## 10.1 启动流程

```text
用户点击“开始游戏”
        ↓
检查游戏 EXE 是否存在
        ↓
检查是否需要启动前同步存档
        ↓
如果需要，检查云端 latest.zip
        ↓
如果云端较新，提示用户是否下载
        ↓
启动游戏 EXE
        ↓
记录 startTime
        ↓
监听游戏进程
        ↓
游戏退出
        ↓
记录 endTime
        ↓
计算 duration
        ↓
写入 play_sessions
        ↓
更新 games.total_play_seconds
        ↓
更新 games.last_launch_time
        ↓
同步游玩记录到 WebDAV
        ↓
根据设置决定是否上传存档
```

------

## 10.2 启动游戏示例代码

```csharp
public async Task LaunchGameAsync(Game game)
{
    if (!File.Exists(game.ExecutablePath))
    {
        throw new FileNotFoundException("游戏启动文件不存在", game.ExecutablePath);
    }

    var startInfo = new ProcessStartInfo
    {
        FileName = game.ExecutablePath,
        Arguments = game.LaunchArgs ?? string.Empty,
        WorkingDirectory = string.IsNullOrWhiteSpace(game.WorkingDirectory)
            ? Path.GetDirectoryName(game.ExecutablePath)!
            : game.WorkingDirectory,
        UseShellExecute = true
    };

    var startTime = DateTimeOffset.Now;

    using var process = Process.Start(startInfo);

    if (process == null)
    {
        throw new InvalidOperationException("游戏启动失败");
    }

    await process.WaitForExitAsync();

    var endTime = DateTimeOffset.Now;
    var duration = endTime - startTime;

    await _playTimeService.RecordSessionAsync(
        game.Id,
        startTime,
        endTime,
        (long)duration.TotalSeconds,
        process.ExitCode
    );
}
```

## 10.3 启动器问题

有些游戏不是直接启动主游戏进程，而是先启动一个 launcher，例如：

```text
launcher.exe
    ↓
real_game.exe
```

这种情况下，`launcher.exe` 可能很快退出，但游戏还在运行。

为了解决这个问题，需要允许用户配置：

```text
启动文件：launcher.exe
实际监控进程：real_game.exe
```

启动流程变成：

```text
启动 launcher.exe
        ↓
等待 real_game.exe 出现
        ↓
开始统计游玩时间
        ↓
监听 real_game.exe 退出
        ↓
结束统计
```

第一版可以先简单处理：

```text
默认监听启动的 EXE 进程。
如果用户发现统计不准确，再手动配置监控进程名。
```

# 11. 存档同步设计

## 11.1 不建议逐文件同步

游戏存档目录可能包含大量小文件，如果直接通过 WebDAV 一个一个上传，会有这些问题：

- 请求数量太多
- 上传速度慢
- 容易失败
- 容易触发 WebDAV 服务限制
- 目录遍历成本高
- 冲突处理复杂

因此第一版建议：

```text
把整个存档目录打包成 zip，然后上传 latest.zip。
```

------

## 11.2 存档同步流程

### 启动游戏前

```text
1. 获取云端 save_manifest.json
2. 比较云端更新时间和本地存档更新时间
3. 如果云端更新：
   - 提示用户是否下载云端存档
   - 下载 latest.zip
   - 备份当前本地存档
   - 解压 latest.zip 到本地存档目录
4. 如果本地更新：
   - 可以选择先上传本地存档
5. 继续启动游戏
```

------

### 游戏退出后

```text
1. 扫描本地存档目录
2. 计算文件列表和 SHA256
3. 与上一次 save_manifest 对比
4. 如果存档没有变化：
   - 不上传
5. 如果存档发生变化：
   - 将存档目录压缩为 latest.zip
   - 上传 latest.zip 到 WebDAV
   - 上传 save_manifest.json
   - 可选：保留一个带时间戳的历史存档 zip
```

## 11.4 文件哈希扫描

用于判断存档是否变化。

```csharp
public async Task<List<SaveFileEntry>> ScanSaveDirectoryAsync(string savePath)
{
    var result = new List<SaveFileEntry>();

    foreach (var file in Directory.GetFiles(savePath, "*", SearchOption.AllDirectories))
    {
        var relativePath = Path.GetRelativePath(savePath, file);
        var fileInfo = new FileInfo(file);

        var sha256 = await ComputeSha256Async(file);

        result.Add(new SaveFileEntry
        {
            RelativePath = relativePath,
            Size = fileInfo.Length,
            Sha256 = sha256,
            LastWriteTime = fileInfo.LastWriteTimeUtc
        });
    }

    return result;
}
```

# 13. 同步冲突处理

## 13.1 为什么会有冲突

例如用户有两台电脑：

```text
台式机 A
笔记本 B
```

可能出现这种情况：

```text
1. A 电脑晚上 8 点玩了游戏，存档变化，但还没上传。
2. B 电脑晚上 9 点也玩了游戏，并上传了存档。
3. A 电脑晚上 10 点再上传存档。
```

这时如果直接覆盖，B 电脑的游戏进度可能丢失。

------

## 13.2 元数据冲突处理

游玩时长不要简单同步一个 `totalPlaySeconds`。

更推荐同步：

```text
每一次游玩记录 play_session
```

每次游戏结束后生成一个游玩记录：

```json
{
  "id": "session-uuid-1",
  "gameId": "game-uuid-1",
  "machineId": "desktop-home",
  "startTime": "2026-05-05T20:15:00-07:00",
  "endTime": "2026-05-05T21:30:00-07:00",
  "durationSeconds": 4500
}
```

云端路径：

```text
games/game-uuid-1/play_sessions/desktop-home_20260505_201500.json
```

多设备同步时，把所有 session 合并，然后求和：

```text
total_play_seconds = 所有 play_session.duration_seconds 的总和
```

这样不会因为多设备覆盖导致游玩时长丢失。

------

## 13.3 存档冲突处理

游戏存档通常无法自动合并，所以不要尝试复杂合并。

推荐策略：

```text
只检测冲突，不自动合并。
让用户选择使用本地还是云端。
```

冲突提示示例：

```text
发现存档冲突：

本地存档：
更新时间：2026-05-05 21:30
大小：12.3 MB

云端存档：
更新时间：2026-05-05 20:10
大小：11.8 MB

请选择：
[使用本地存档并上传]
[使用云端存档并覆盖本地]
[先备份两份，再让我选择]
[取消]
```

------

## 13.4 存档冲突判断逻辑

可以使用：

```text
本地 save_manifest
云端 save_manifest
上一次同步时的 save_manifest
```

判断逻辑：

```text
如果本地 == 上次同步，云端 != 上次同步：
    云端有更新，可以下载

如果本地 != 上次同步，云端 == 上次同步：
    本地有更新，可以上传

如果本地 != 上次同步，云端 != 上次同步：
    本地和云端都变了，发生冲突

如果本地 == 云端：
    不需要同步
```

```text
计算本次游玩时长
        ↓
写入 play_sessions
        ↓
更新 games 表
        ↓
同步 play_session 到云端
        ↓
如果开启退出后同步：
            扫描存档目录
            如果存档有变化：
                打包 latest.zip
                上传 WebDAV
                上传 save_manifest.json
        ↓
刷新 UI
```

------

# 17. 存档上传流程

```text
开始上传存档
        ↓
检查存档目录是否存在
        ↓
扫描存档目录
        ↓
计算文件 hash
        ↓
读取上次本地 save_manifest
        ↓
判断是否有变化
        ↓
如果没有变化：
    结束，不上传
        ↓
如果有变化：
    创建本地备份
        ↓
压缩存档目录为 latest.zip
        ↓
上传到 WebDAV:
    games/{gameId}/saves/latest.zip
        ↓
可选上传历史版本:
    games/{gameId}/saves/save_yyyyMMdd_HHmmss.zip
        ↓
上传 save_manifest.json
        ↓
写入 sync_records
        ↓
完成
```

# 18. 存档下载流程

```text
开始下载云端存档
        ↓
检查云端 latest.zip 是否存在
        ↓
下载到本地 temp/downloads
        ↓
备份当前本地存档目录
        ↓
清空或覆盖本地存档目录
        ↓
解压 latest.zip 到存档目录
        ↓
更新本地 save_manifest
        ↓
写入 sync_records
        ↓
完成
```

# 20. 路径和数据分离原则

这是项目架构中非常重要的一点。

## 20.1 云端同步的数据

适合同步到云端的数据：

```text
游戏 ID
游戏名称
游戏封面
游玩记录
总游玩时长
上次启动时间
存档压缩包
存档 manifest
同步日志
```

------

## 20.2 不适合全局同步的数据

不适合作为全局数据直接同步的数据：

```text
游戏 EXE 绝对路径
游戏安装目录绝对路径
游戏存档目录绝对路径
启动参数
实际监控进程名
```

这些信息应该按设备保存。

原因是不同电脑上的路径可能完全不同。

------

## 20.3 正确做法

全局游戏数据：

```text
games/{gameId}/metadata.json
```

本机路径数据：

```text
games/{gameId}/paths/{machineId}.json
```

这样可以实现：

```text
同一个游戏，在多台电脑上共享封面、名称、游玩时长和存档。
但每台电脑有自己的 EXE 路径和存档路径。
```

# 21. 安全设计

## 21.1 WebDAV 密码保存

不要明文保存密码。

推荐方案：

```text
Windows Credential Manager
```

或者：

```text
DPAPI 加密后保存
```

不要这样做：

```json
{
  "webdavPassword": "123456"
}
```

推荐保存：

```text
config.json 只保存 webdav_url 和 username
密码单独交给 Windows Credential Manager 保存
```

------

## 21.2 存档覆盖保护

任何时候只要要覆盖本地存档，都必须先备份。

例如：

```text
下载云端 latest.zip 并解压之前
解决冲突并覆盖之前
用户手动恢复历史存档之前
```

------

## 21.3 删除保护

删除游戏时不要默认删除本地游戏文件和存档文件。

删除游戏应该只删除管理器里的记录：

```text
默认行为：
只从游戏管理器移除，不删除游戏本体和存档。

高级选项：
允许用户手动勾选是否删除封面缓存、备份文件、云端数据。
```

# 27. 推荐最终技术方案总结

最终推荐方案：

```text
C# / .NET 8 + WPF + SQLite + WebDAV
```

整体架构：

```text
本地优先：
- SQLite 保存游戏库
- AppData 保存封面缓存、备份、日志
- 本地可以完全离线使用

云端同步：
- WebDAV 保存 metadata.json
- WebDAV 保存 cover.jpg
- WebDAV 保存 play_session 记录
- WebDAV 保存 save_manifest.json
- WebDAV 保存 latest.zip 存档包

游戏启动：
- 使用 Process.Start 启动 EXE
- 使用进程监听统计游玩时长
- 游戏退出后更新数据库和云端记录

存档同步：
- 不逐文件同步
- 将存档目录打包成 zip
- 上传 latest.zip
- 覆盖前必须本地备份
- 多设备冲突时让用户手动选择
```

# 28. 一句话架构总结

```text
这个项目应该设计为一个 Windows 本地优先的游戏库管理器：使用 C#/.NET 8 + WPF 构建桌面 UI，SQLite 保存本地游戏数据，Process 管理游戏启动和时长统计，WebDAV 负责同步游戏元数据、封面、游玩记录和存档压缩包，存档同步采用 zip 打包加 manifest 校验，并通过本地备份和冲突提示保证用户存档安全。
```
