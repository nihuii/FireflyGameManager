## 5. 阶段 1：外部资料 WebDAV 同步

### 5.1 当前缺口

当前本地 SQLite 已有 `game_external_metadata` 表，`Game` 也能携带 `ExternalGameMetadata`。但是 WebDAV V2 的 `GameCloudMetadata` 只包含游戏 ID、名称、封面文件名、游玩时长、最近启动时间、更新时间和游玩会话 ID，没有外部资料快照。手动整库上传还会清空 `game_external_metadata`，说明外部资料目前没有真正进入云同步闭环。

### 5.2 推荐云端结构

使用独立文件保存外部资料，不修改现有 `metadata.json` 的职责：

```text
v2/
  games-index.json
  games/
    {gameId}/
      metadata.json
      external-metadata.json
      cover/
      paths/
      play-sessions/
      saves/
```

`external-metadata.json` 建议格式：

```json
{
  "schemaVersion": 1,
  "gameId": "local-game-id",
  "provider": "bangumi",
  "subjectId": "123456",
  "isLinked": true,
  "originalName": "Original title",
  "localizedName": "本地化标题",
  "summary": "资料简介",
  "releaseDate": "2024-01-01",
  "developer": "Developer",
  "publisher": "Publisher",
  "tags": ["ADV", "视觉小说"],
  "imageUrl": "https://...",
  "subjectUrl": "https://bangumi.tv/subject/123456",
  "sourceUpdatedAtUtc": "2026-06-17T00:00:00Z",
  "snapshotUpdatedAtUtc": "2026-06-17T00:00:00Z"
}
```

`snapshotUpdatedAtUtc` 表示 Firefly 本地资料快照更新时间，用于设备间冲突判断；`sourceUpdatedAtUtc` 表示上游资料源的更新时间，继续保留原语义。

### 5.3 代码改造点

1. 新增模型：
   - `GameManager.App/Models/ExternalGameMetadataCloudSnapshot.cs`
   - 字段包含 `SchemaVersion`、`GameId`、`Metadata` 或扁平化资料字段、`SnapshotUpdatedAtUtc`。
2. 扩展本地库接口：
   - `IGameLibraryService.GetExternalMetadataSnapshot(string gameId)`
   - `IGameLibraryService.GetExternalMetadataSnapshots()`
   - `IGameLibraryService.ApplyCloudExternalMetadata(ExternalGameMetadataCloudSnapshot snapshot)`
3. 扩展 SQLite 服务：
   - 从 `game_external_metadata.updated_at` 读取 Firefly 快照更新时间。
   - 写入云端资料时保留或推进 `updated_at`，避免下载后立即被本机视为更旧。
4. 扩展 WebDAV V2 接口：
   - `IWebDavGameSyncService.UploadExternalMetadataAsync(...)`
   - `IWebDavGameSyncService.DownloadExternalMetadataAsync(...)`
   - `IWebDavGameSyncService.DeleteExternalMetadataAsync(...)` 可选，用于用户明确删除资料时清理云端。
5. 改造 `WebDavGameSyncService.UploadGameAsync`：
   - 上传基础 `metadata.json` 后，如果本地存在 `ExternalGameMetadata`，上传 `external-metadata.json`。
   - 如果本地明确解除关联但保留快照，仍上传 `isLinked=false` 的资料快照。
   - 不上传 `BangumiAccount`、Access Token、`BangumiCollectionState`。
6. 改造 `WebDavCloudMetadataPullService` 和 `WebDavFullSyncService`：
   - 下载基础游戏元数据后，按游戏 ID 拉取 `external-metadata.json`。
   - 本地没有资料时直接应用云端资料。
   - 本地和云端是同一 provider + subject 时，按 `snapshotUpdatedAtUtc` 选择较新版本。
   - 本地和云端 subject 不一致时，不自动覆盖，记录冲突。
7. 保持 `WebDavManualSyncService` 的安全策略：
   - 旧版整库上传继续排除 `game_external_metadata`。
   - 阶段 C 外部资料只通过 V2 的 `external-metadata.json` 同步。

### 5.4 冲突策略

| 场景 | 行为 |
| --- | --- |
| 本地无资料，云端有资料 | 下载并写入本地 |
| 本地有资料，云端无资料 | 下次上传本地资料 |
| provider + subject 相同，云端较新 | 用云端覆盖本地快照 |
| provider + subject 相同，本地较新或相同 | 保留本地，下次上传本地 |
| provider 相同但 subject 不同 | 不自动覆盖，记录冲突 |
| provider 不同 | 不自动覆盖，等待多资料源阶段处理 |
| 云端 JSON 解析失败 | 跳过该游戏并写同步日志 |

### 5.5 测试计划

1. 上传游戏时会生成 `v2/games/{gameId}/external-metadata.json`。
2. 上传请求体不包含 Access Token、Refresh Token、用户名私密收藏缓存。
3. 下载到另一台设备时，`ExternalGameMetadata` 可以离线显示。
4. 云端较旧资料不会覆盖本地较新资料。
5. 云端较新资料会覆盖本地旧资料。
6. subject 不一致时不覆盖，并产生同步日志。
7. 旧版整库上传仍会排除 `game_external_metadata` 和 `bangumi_collection_states`。
8. 完整测试和构建通过。

## 6. 阶段 2：外部资料冲突处理与 UI

### 6.1 目标

阶段 1 可以先采用保守策略记录冲突；阶段 2 需要让用户在界面上看见并解决冲突，避免资料长期卡住。

### 6.2 实现方案

1. 新增冲突模型：
   - `GameManager.App/Models/ExternalMetadataConflict.cs`
   - 保存 `GameId`、本地资料、云端资料、发现时间、冲突原因。
2. 新增本地持久化：
   - 可以先复用 SQLite，新增 `external_metadata_conflicts` 表。
   - 或者先写入同步日志，并在详情页按需重新检测。推荐 SQLite 表，避免用户错过启动时的冲突。
3. 详情页 UI：
   - 在游戏详情页外部资料区域显示冲突提示。
   - 提供“使用本机资料”“使用云端资料”“取消关联并保留本机快照”三个操作。
   - 不提供自动合并字段，避免把不同条目的简介、标签、封面混在一起。
4. 同步日志：
   - `sync_records` 增加 `external-metadata` 类型记录。
   - 日志消息不包含 Token，不包含完整长简介，只保留 provider、subject、游戏名和结果。

### 6.3 验收标准

1. 用户能在详情页看见资料冲突。
2. 选择本机或云端后，冲突记录消失。
3. 解决结果会在下一次 WebDAV 同步时上传。
4. UI 不遮挡现有启动、备份、收藏状态操作。

## 7. 阶段 3：Bangumi 批量与收藏体验补全

### 7.1 批量匹配本地游戏

目标是在已有单游戏搜索导入基础上，增加“待匹配列表”，减少大量游戏库首次整理成本。

实现方案：

2. 对每个游戏使用现有 `IGameMetadataProvider.SearchAsync` 查询候选。
3. 只自动预选高置信结果，不自动写入。
4. 用户逐项确认或跳过。
5. 批量应用时复用现有字段导入策略，新游戏默认导入名称和封面，已有游戏默认保守导入简介、日期、厂商、标签。

1. 批量匹配不会阻塞手动编辑。
2. 搜索失败的游戏显示失败原因，可以重试。

3. 保存时复用 `BangumiCollectionState` 和 `BangumiApiClient.SaveCollectionAsync`。
4. API 不支持或失败时只影响收藏区，不影响本地详情。

验收标准：

1. 可以读取并修改收藏状态、评分、评论。
2. 退出 Bangumi 账号后，本地已导入资料继续显示。
3. 授权失效时账号进入 `RequiresReconnect`，不会重复请求失败接口。

### 7.3 搜索体验继续优化

## 9. 阶段 5：安全 OAuth 快捷登录

### 9.1 当前限制

桌面客户端不能内置共享 OAuth `client_secret`。如果直接把生产 Secret 放进 WPF 应用，反编译即可提取，风险高于当前 Access Token 方案。

### 9.2 推荐顺序

1. 保留 Access Token 登录作为稳定入口。
2. 优先调研 Bangumi 是否有适合公共客户端的 PKCE 流程。
3. 若没有 PKCE，则二选一：
   - 让高级用户自行填写 OAuth Client ID 和 Secret。
   - 部署 Firefly 自有 HTTPS OAuth 中转服务，由服务端保存 Secret。

### 9.3 客户端实现方案

1. 新增 OAuth 模型：
   - `BangumiOAuthSettings`
   - `BangumiOAuthTokenResponse`
   - `BangumiOAuthSession`
2. 新增服务：
   - `IBangumiOAuthClient`
   - `BangumiOAuthClient`
   - `LoopbackOAuthCallbackListener` 或 `OAuthRelayClient`
3. 账号存储：
   - Access Token、Refresh Token、过期时间全部 DPAPI 加密。
   - Refresh Token 不导出，不同步，不写日志。
4. UI：
   - 设置页保留“使用 Access Token 连接”。
   - 增加“浏览器授权登录”，并标注需要安全中转或用户自有 OAuth 应用。
5. Token 刷新：
   - API 调用前若即将过期，先刷新。
   - 刷新失败后设置 `RequiresReconnect=true`。

### 9.4 验收标准

1. 浏览器登录不会暴露生产 Secret。
2. Token 刷新失败不会破坏本地游戏库。
3. WebDAV、导出、日志中都没有 OAuth Token。

### 10.1 目标

### 10.2 Provider 架构

1. 保留并扩展 `IGameMetadataProvider`：
   - `ProviderId`
   - `DisplayName`
   - `RequiresApiKey`
   - `SearchAsync`
   - `GetDetailsAsync`
2. 新增 `IGameMetadataProviderRegistry`：
   - 注册 Bangumi、SteamGridDB、RAWG、VNDB、IGDB。
   - 设置页控制启用、排序、API Key。
3. 缓存隔离：
   - 每个 provider 独立搜索缓存。
   - 图片缓存目录按 provider 分隔。
4. 数据模型：
   - `ExternalGameMetadata.Provider` 继续作为资料源标识。
   - 不同 provider 的 subject ID 不互通。
5. UI：
   - 添加/编辑游戏时可选择资料源。
   - 默认优先 Bangumi，后续按用户启用顺序搜索。

### 10.3 推荐接入顺序

1. SteamGridDB：优先做封面和背景图补充，资料字段少，和现有封面能力契合。
2. RAWG：覆盖通用 PC 游戏资料，适合作为非视觉小说资料源。
3. VNDB：适合视觉小说，但字段和分类需要单独映射。
4. IGDB：资料丰富，但鉴权和字段复杂度更高，放在后面。

### 10.4 验收标准

1. Bangumi 现有行为不回退。
2. 每个资料源失败时只影响自身，不影响其他资料源。
3. API Key 加密保存，不进入导出、WebDAV、日志。
4. 阶段 C 的外部资料同步能携带 provider 信息。

## 11. 阶段 7：打包与发布流程

### 11.1 目标

把开发态 WPF 项目整理为用户可安装、可升级、可排查问题的 Windows 桌面应用。

### 11.2 实现方案

1. 增加发布脚本：
   - `scripts/publish.ps1`
   - 生成 self-contained 或 framework-dependent 包。
2. 选择安装器：
   - 初期推荐 Inno Setup，脚本简单，适合个人项目。
   - 如果需要商店分发或企业部署，再评估 MSIX。
3. 版本信息：
   - 在 `.csproj` 中维护 `Version`、`FileVersion`、`AssemblyVersion`。
   - 设置页显示当前版本。
4. 发布产物：
   - 安装包。
   - 便携 zip 包。
   - 更新说明。
5. 升级兼容：
   - 不删除 `%LocalAppData%\FireflyGameManager`。
   - 保留 SQLite、封面缓存、备份、设置、账号文件。

### 11.3 验收标准

1. 全新安装后可以启动并创建本地数据目录。
2. 覆盖安装不会丢失本地游戏库、账号状态和备份。
3. 卸载时默认不删除用户数据，除非用户明确选择。

## 12. 阶段 8：分类、统计、多云盘

### 12.1 游戏分类

实现方案：

1. SQLite 增加本地标签、平台、收藏状态、安装来源字段。
2. 游戏库页面增加筛选和分组。
3. Bangumi/RAWG/VNDB 标签只作为建议，不自动覆盖用户标签。

验收标准：

1. 用户可以按自定义标签、平台、最近游玩筛选。
2. 删除外部资料不会删除用户本地分类。

### 12.2 统计面板

实现方案：

1. 基于 `play_sessions` 统计今日、本周、本月、总游玩时长。
2. 增加最近启动记录和排行榜。
3. 统计页只读，不直接修改游戏数据。

验收标准：

1. 跨设备同步后的 play session 不重复计数。
2. 时区变化不会导致明显统计错误。

### 12.3 多云盘

当前代码以 WebDAV 服务为主，长期若要支持 OneDrive、Google Drive、Dropbox、SFTP 或本地 NAS，应先抽象通用云存储接口。

实现方案：

1. 新增 `ICloudStorageClient`：
   - `GetJsonAsync`
   - `PutJsonAsync`
   - `GetFileAsync`
   - `PutFileAsync`
   - `ListAsync`
   - `EnsureDirectoryAsync`
2. 让 `WebDavGameSyncService` 先通过 WebDAV 适配器实现该接口。
3. 后续每个云盘只实现存储适配器，复用 V2 同步协议。

验收标准：

1. WebDAV 行为不回退。
2. 新云盘不会复制一套同步业务逻辑。
3. 云端目录结构仍兼容 Firefly V2。
